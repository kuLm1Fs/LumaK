# 如何避免做成玩具项目

## 会显得像玩具的信号

1. 只有 happy path
2. 没日志
3. 没测试
4. 只有 UI，没有 runtime 设计
5. 只有 prompt，没有边界和约束
6. 没有评估，只会说“感觉效果不错”

## 会显得不像玩具的信号

1. 有清晰分层
   - interface
   - runtime
   - tools
   - storage

2. 有执行轨迹
   - 模型如何决策
   - 调了什么工具
   - 返回了什么

3. 有 guardrails
   - schema
   - validate
   - 路径限制
   - timeout
   - 错误分类

4. 有评估
   - 至少 10 到 20 个任务样例
   - 成功标准
   - 失败分析

5. 有设计文档
   - 为什么这样拆层
   - 为什么先做单 agent
   - 为什么故意不做某些高级特性

## 简历上应该怎么写

不要写：

- 做了一个 AI demo
- 接入了 OpenAI API
- 做了一个智能代码助手

应该写：

- 实现了一个带工具调用闭环的本地代码库 agent runtime
- 为工具层加入参数校验、工作目录隔离、超时与错误分类机制
- 构建任务评估样例集，用于验证工具选择和任务完成质量

## GitHub 上应该具备什么

至少要有：

- `README.md`
- `docs/architecture.md` 或同类说明
- `tests/`
- `eval/` 或评估文档
- 一张架构图
- 一段执行轨迹示例

## 面试时要能讲清的事

1. 这个项目为什么是 agent，而不是普通聊天壳子
2. 工具调用闭环怎么实现
3. 为什么要做路径限制和参数校验
4. 如果工具失败或模型输出错了怎么办
5. 你怎么评估它不是“看起来聪明”而是真的有用

## 最后的判断标准

完成后，用这 6 个问题审判项目：

1. 去掉 LLM 后还剩什么
2. 模型输出错了系统会怎样
3. 有没有简单 baseline 或任务评测
4. 最 risky 的设计是什么
5. 别人是否能复现运行
6. 你能不能不看代码讲 10 分钟架构

如果这 6 个问题里答不上来 3 个以上，项目大概率还是偏玩具。
