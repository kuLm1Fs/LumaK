# 项目完成路线

## 目标

在 7 天内完成一个可运行、可展示、可测试、可讲解的“本地代码库理解与安全修改 Agent”，并让它具备写进简历和应对面试追问的基础质量。

这个路线的核心原则只有一条：

- 不追求大而全，只追求把最小闭环做扎实

## 第一版边界

第一版只做下面 4 个工具：

- `list_files`
- `read_file`
- `search_text`
- `safe_edit`

第一版只支持：

- 本地工作目录内的代码仓库
- 文本文件读取与修改
- 单轮到多轮的小任务
- CLI 交互

第一版明确不做：

- 多 agent
- 前端界面
- 向量数据库或复杂 RAG
- 浏览器自动化
- 插件系统
- 长任务调度
- 复杂 memory

## 总体实施顺序

按下面的顺序推进，不要乱序扩展：

1. 项目骨架和最小 CLI
2. 模型接入
3. `list_files`、`read_file`、`search_text`
4. 最小 tool-calling loop
5. 路径限制、参数校验、超时和错误分类
6. 执行轨迹和日志
7. `safe_edit`
8. session/history
9. 测试
10. 评测任务、README、架构图、demo

## 阶段路线

### 阶段 1：搭最小可运行骨架

目标：

- 建立目录结构
- 做最小 CLI
- 接通一个模型
- 完成一次普通问答

完成标准：

- 命令行可以启动
- 可以输入问题并拿到回答
- README 里有最小启动说明

这一阶段不要做的事：

- 不要先做复杂框架抽象
- 不要先做前端
- 不要先做 fancy prompt 工程

### 阶段 2：跑通工具调用闭环

目标：

- 实现 `list_files`
- 实现 `read_file`
- 实现 `search_text`
- 让模型可以决定是否调用工具
- 工具结果可以回喂模型继续推理

完成标准：

- 用户可以问“入口在哪”
- 用户可以问“某模块负责什么”
- 用户可以问“和某功能相关的代码在哪”
- 最终回答不是简单拼接原始工具输出

这一阶段的关键不是工具数量，而是闭环：

1. 用户提问
2. runtime 判断并发起模型请求
3. 模型选择工具
4. runtime 执行工具
5. 工具结果回喂模型
6. 模型给出最终回答

### 阶段 3：补安全边界

目标：

- 限制工具只能访问工作目录
- 对工具参数做校验
- 加入超时控制
- 统一错误类型和错误提示

完成标准：

- 越界路径会被拦截
- 非法参数会被拒绝
- 文件不存在不会导致程序直接崩溃
- 工具失败会返回可理解的错误

这是项目成色的关键阶段，因为它决定系统是不是“受约束的 agent runtime”，而不是只会在 happy path 上工作。

### 阶段 4：补可观测性

目标：

- 记录执行轨迹
- 保存日志
- 明确区分 `user`、`assistant`、`tool` 消息

至少记录：

- `session_id`
- 用户问题
- 调用的工具名
- 工具参数
- 成功或失败
- 执行耗时
- 最终回答

完成标准：

- 可以查看一轮任务完整执行过程
- 可以用轨迹解释 agent 为何得出某个回答

### 阶段 5：做安全修改能力

目标：

- 实现 `safe_edit`
- 提供修改前 diff 预览
- 拒绝危险或非法修改

`safe_edit` 至少应包含这些约束：

- 只能写工作目录内文件
- 只允许处理文本文件
- 写前校验原始内容是否匹配
- 输出明确 diff
- 失败时不静默写入

完成标准：

- 可以修改示例文件
- 越界写入会被拒绝
- 原文不匹配时不会硬写覆盖

### 阶段 6：补状态和多轮

目标：

- 保存会话历史
- 支持同一个 session 的连续追问
- 让 agent 能在多轮里利用前文

完成标准：

- 同一 session 至少能连续问两到三轮
- 历史消息结构清晰
- tool 调用记录和回答记录可回放

### 阶段 7：补证据链

目标：

- 工具单测
- 至少 2 个 agent flow 测试
- 准备 10 到 20 个评测任务
- 准备失败案例
- 在一个真实仓库上完成 demo

完成标准：

- 不只“感觉可用”
- 而是“有测试、有任务集、有失败样例”

推荐的评测任务类型：

- 找入口文件
- 解释模块职责
- 搜索与某功能相关的代码
- 总结局部架构
- 修改一个小逻辑或配置
- 拒绝越界修改

推荐的失败案例：

- 搜索无结果
- 文件不存在
- 路径越界
- 参数错误
- 修改原文不匹配
- 写入被拒绝

## 每天的优先级原则

每天都按下面这个顺序排优先级：

1. 先保证闭环能跑
2. 再补边界和错误处理
3. 再补日志和轨迹
4. 最后才补包装材料

如果时间不够，优先保留：

- 工具闭环
- 安全边界
- 执行轨迹
- 测试和评测

如果时间不够，优先砍掉：

- 前端
- 高级规划能力
- 多 agent
- 复杂 memory
- 所有不影响面试展示核心论点的装饰项

## 项目完成标准

只有满足下面这些条件，项目才算真正完成：

1. 能回答本地代码库问题
2. 能进行多步工具调用
3. 能拒绝非法路径和危险修改
4. 能记录完整执行轨迹
5. 有基础测试
6. 有评测任务和失败样例
7. 有 README、架构说明和 demo

## 最后判断标准

项目最终要能支撑你讲清楚这几件事：

- 为什么这是 agent，而不是普通问答壳
- 工具调用闭环如何实现
- 为什么需要路径限制和参数校验
- 修改能力为什么可以叫“安全修改”
- 系统失败时如何处理
- 你如何评估它是否真的有用

如果这些问题里有一半讲不清，说明项目还没真正完成。
