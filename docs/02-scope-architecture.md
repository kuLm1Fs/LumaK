# 功能范围与架构

## 项目定位

这是一个“带工具调用闭环的本地代码库 Agent”。

它的第一版目标不是做全能开发助手，而是把下面这几个高价值能力做扎实：

- 看目录
- 读文件
- 搜索代码
- 基于上下文回答问题
- 在受限条件下修改文件

## 第一版功能范围

### 核心能力

1. `list_files`
   - 查看目录结构
   - 支持限制根目录
   - 可选递归、深度限制

2. `read_file`
   - 读取文本文件
   - 支持行号范围
   - 限制读取目录范围

3. `search_text`
   - 搜索关键词、函数名、类名、TODO
   - 返回命中位置和上下文片段

4. `answer_with_context`
   - 基于工具结果回答问题
   - 汇总而不是机械拼接工具输出

5. `safe_edit`
   - 在受限范围内修改文件
   - 最好提供 diff 预览
   - 失败时给明确提示

## 非目标

第一版不要做：

- 多 agent
- RAG 全家桶
- 浏览器自动化
- 插件系统
- 复杂 memory
- 长任务调度
- 复杂前端工作台

这些都很诱人，但现在加入只会提高噪音、降低完成质量。

## 推荐分层

### 1. Interface 层

负责：

- CLI 输入输出
- 后续 Web API / Web UI 接入点

不负责：

- 直接实现工具逻辑
- 直接操作模型上下文

### 2. Agent Runtime 层

负责：

- 用户消息进入
- 构建 prompt / messages
- 决定是否执行工具
- 执行多步 tool loop
- 汇总最终结果

不负责：

- 工具底层实现
- 文件系统直接读写

### 3. Tool 层

负责：

- schema
- 参数校验
- 路径限制
- 具体执行
- 错误归类

不负责：

- 大模型推理流程

### 4. Storage / Trace 层

负责：

- 会话历史
- 工具调用轨迹
- 日志
- 评估记录

不负责：

- 直接做决策

## 最小运行闭环

1. 用户提问
2. agent 判断是否需要工具
3. 模型选择工具
4. runtime 校验并执行工具
5. 工具结果回喂模型
6. 模型生成最终回答
7. 记录轨迹和日志

## 需要体现的工程点

### 安全边界

- 只能访问工作目录
- 工具参数必须校验
- 修改类工具必须更谨慎

### 失败处理

- 工具超时
- 文件不存在
- 非法路径
- 空结果
- 模型错误输出

### 可观测性

至少记录：

- session_id
- 用户问题
- 调用了哪些工具
- 工具参数
- 执行耗时
- 成功或失败
- 最终回答

### 测试

至少覆盖：

- 工具层单测
- 路径越界测试
- 一个完整的 tool-calling 流程测试

## 为什么这个方向能自然接 TS

后续如果想补 TS，可以直接为它加一个轻量前端：

- 左侧是代码库或会话列表
- 中间是对话界面
- 右侧是执行轨迹和工具调用记录

这样你的项目会从“会做 agent runtime”自然演进到“会做 AI 应用交付”。
